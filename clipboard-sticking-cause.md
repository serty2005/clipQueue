# Причина залипания буфера при копировании изображений

## Короткий вывод

Основная причина залипания находится в автоматическом чтении изображений из системного буфера сразу после `WM_CLIPBOARDUPDATE`.

ClipQueue после каждого изменения буфера сам открывает буфер и пытается немедленно вычитать изображение в `CF_DIBV5`, а при неудаче повторно открывает буфер и пытается читать `CF_DIB`:

- `main.go:231-262`
- `internal/app/controller.go:146-166`
- `platform/windows/clipboard.go:93-181`

Для изображений это самый тяжёлый путь, потому что приложение не просто проверяет факт изменения буфера, а синхронно запрашивает у Windows полный bitmap-представление и копирует его целиком в память, пока буфер остаётся открытым:

- `platform/windows/clipboard.go:58-83`

Именно из-за этого буфер может выглядеть как "залипший": пока наш процесс держит `OpenClipboard` и ждёт/копирует DIB, другие приложения не могут нормально работать с буфером.

## Что именно происходит в коде

1. Системное окно получает `WM_CLIPBOARDUPDATE`, после чего событие уходит в фонового воркера.
2. Воркер вызывает `controller.OnClipboardUpdate()`, а тот почти сразу вызывает `windows.Read()`.
3. `windows.Read()` открывает буфер и при наличии формата изображения запрашивает сначала `CF_DIBV5`, потом при fallback ещё и `CF_DIB`.
4. `readClipboardDIBBytes()` делает `GetClipboardData`, `GlobalLock`, `GlobalSize` и полное копирование DIB-данных в Go-срез до закрытия буфера.

Это особенно плохо именно для изображений, потому что:

- DIB занимает много памяти;
- Windows может синтезировать нужный формат по требованию;
- часть приложений публикует изображения через delayed rendering, то есть реальная генерация данных начинается только в момент `GetClipboardData`.

## Почему симптом проявляется именно как "буфер перестаёт отзываться"

По документации WinAPI приложение, запросившее delayed-rendered формат, держит буфер открытым в момент, когда владелец рендерит данные:

- `WM_RENDERFORMAT`: https://learn.microsoft.com/en-us/windows/win32/dataxchg/wm-renderformat
- `GetClipboardData`: https://learn.microsoft.com/en-us/windows/win32/api/winuser/nf-winuser-getclipboarddata

Критичный момент: в `WM_RENDERFORMAT` прямо сказано, что владелец не может сам открыть буфер, потому что в этот момент буфер уже удерживается приложением, которое запросило формат.

Отсюда и наблюдаемое поведение:

- пользователь копирует изображение;
- ClipQueue почти сразу лезет в `CF_DIBV5/CF_DIB`;
- если источник изображения отдаёт bitmap лениво или через дорогую конвертацию, наш процесс надолго удерживает буфер;
- новые copy/paste начинают выглядеть зависшими, пока операция не завершится или процесс ClipQueue не будет закрыт.

## Почему именно текущая реализация усиливает проблему

- Код не выбирает "первый пригодный формат", а агрессивно форсирует `CF_DIBV5`, затем `CF_DIB`.
- Для одного и того же изображения возможны две тяжёлые попытки чтения подряд.
- Полное копирование DIB выполняется до `CloseClipboard`, то есть всё это время буфер недоступен для остальных.

С точки зрения WinAPI это особенно рискованно, потому что система поддерживает синтез форматов по требованию:

- `GetClipboardData`: https://learn.microsoft.com/en-us/windows/win32/api/winuser/nf-winuser-getclipboarddata
- `EnumClipboardFormats`: https://learn.microsoft.com/en-us/windows/win32/api/winuser/nf-winuser-enumclipboardformats
- `GetPriorityClipboardFormat`: https://learn.microsoft.com/ru-ru/windows/win32/api/winuser/nf-winuser-getpriorityclipboardformat

## Сопутствующий дефект, который ухудшает восстановление

Есть ещё один опасный участок в записи буфера:

- `platform/windows/clipboard.go:437-462`

Если `SetClipboardData` возвращает `0`, код считает это ошибкой только когда `sysErr` не равен строке `"The operation completed successfully."`.

То есть возможен сценарий, когда `SetClipboardData` уже провалился, но `Write()` всё равно выходит как успешный. После этого приложение может считать, что буфер восстановлен, хотя по факту он уже пустой или в неконсистентном состоянии.

Я не считаю этот дефект первичным триггером именно описанного "залипания" при простом копировании изображения, но он делает восстановление буфера после проблемных операций менее надёжным.

## Итог

Первичный корень проблемы: ClipQueue сам инициирует тяжёлое и потенциально блокирующее чтение bitmap-форматов изображения сразу после `WM_CLIPBOARDUPDATE`, причём делает это в самой дорогой форме `CF_DIBV5 -> CF_DIB`, удерживая буфер открытым на время запроса и копирования данных.

По симптомам это полностью совпадает с поведением пользователя: закрытие утилиты освобождает зависший захват буфера, а повторная запись буфера через интерфейс может временно заменить проблемное содержимое на уже локально сохранённое.
